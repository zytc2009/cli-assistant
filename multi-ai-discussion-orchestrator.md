# Multi-AI Discussion Orchestrator — 设计文档

## 核心理念

用户提出一个想法，多个 AI 各自独立发表观点，用户选定一个 AI 担任**主持人**，在已有信息基础上引导多轮讨论，每轮由各 AI 补充和完善，最终由主持人综合所有观点输出结果文档。

**与现有设计的关键差异**：主持人不再是代码逻辑，而是一个真实的 AI Agent——它理解上下文、能引导讨论方向、能综合各方观点形成最终输出。

## Phase 详解

### Phase 1: 独立发言

**目的**：收集多元化的初始观点，避免先发者锚定后续思考。

**流程**：
1. 用户输入想法（一段文字描述，可以是问题、方案草案、需求等）
2. 系统将想法同时发送给所有参会 AI
3. 各 AI **并行**、**独立**地返回自己的观点
4. 系统展示所有观点的摘要列表

**Prompt 设计**：
```
你是 {agent_name}，擅长 {agent_strengths}。

用户提出了以下想法：
{user_idea}

请从你的专业视角对这个想法发表看法。要求：
1. 对想法的整体评价（优势和潜在问题）
2. 你认为最关键的 2-3 个要点
3. 你的补充建议或替代方案

请独立思考，不要假设其他人的观点。
```

**输出格式**：
```
┌──────────────────────────────────────────────────┐
│ 所有 AI 的初始观点                                 │
├──────────────────────────────────────────────────┤
│ 1. Claude Sonnet    → 关注架构可扩展性，建议…       │
│ 2. Codex            → 关注实现可行性，建议…         │
│ 3. Gemini           → 关注行业趋势对比，建议…       │
│ 4. Kimi             → 关注用户体验，建议…           │
├──────────────────────────────────────────────────┤
│ 请选择主持人 [1-4]：                               │
│ （可选）补充意见：                                  │
└──────────────────────────────────────────────────┘
```

---

### Phase 2: 主持人引导的多轮讨论

**目的**：在主持人的引导下，各 AI 围绕焦点问题深入讨论，逐步完善方案。

**主持人的职责**：
- 总结上一轮讨论的共识与分歧
- 提出本轮需要聚焦的问题
- 引导讨论向可落地的方向收敛
- 确保每个重要观点都被充分讨论

**每轮流程**：

```
Step 1: 主持人开场
   ├── 输入：上一轮所有发言 + 用户补充意见
   ├── 输出：本轮讨论引导语
   │     · 上轮要点总结
   │     · 尚未解决的分歧
   │     · 本轮讨论焦点（1-2 个具体问题）
   └── 作用：为其他 AI 的发言提供方向

Step 2: 各 AI 依次发言
   ├── 顺序执行（每个 AI 能看到主持人引导 + 前面已发言者的内容）
   ├── 输入：用户想法 + 所有历史发言 + 主持人本轮引导
   └── 输出：针对主持人提出的焦点问题进行补充/完善/反驳

Step 3: 展示本轮结果
   ├── 系统展示本轮摘要
   └── 用户选择：
         [c] 继续下一轮
         [f] 补充意见后继续
         [d] 结束讨论，进入 Phase 3
```

**主持人开场 Prompt**：
```
你是本次讨论的主持人 {agent_name}。

讨论主题：{user_idea}

{history_section}

{user_feedback_section}

作为主持人，请：
1. 用 2-3 句话总结上一轮讨论的关键共识
2. 指出尚存的 1-2 个主要分歧或待深入的问题
3. 明确提出本轮讨论的焦点问题，引导大家聚焦

注意：你是引导者，不是裁判。目标是帮助团队充分讨论并逐步收敛，而不是强制统一意见。
```

**参会者发言 Prompt**：
```
你是 {agent_name}，擅长 {agent_strengths}。

讨论主题：{user_idea}

{history_section}

主持人 {moderator_name} 本轮引导：
{moderator_opening}

请针对主持人提出的焦点问题发表你的看法：
1. 对焦点问题的具体回应
2. 补充或完善你之前的观点（如果有新的思考）
3. 对其他参会者观点的回应（同意/反驳/补充）

保持简洁聚焦，避免重复已达成共识的内容。
```

**轮次控制**：
- 默认最大 3 轮（用户可配置）
- 用户随时可以提前结束
- 如果主持人判断已充分讨论，可建议用户进入 Phase 3

---

### Phase 3: 主持人输出结果

**目的**：由主持人综合全部讨论内容，输出一份结构化的结果文档。

**流程**：
1. 将所有轮次的讨论历史 + 用户的所有补充意见传给主持人
2. 主持人生成最终结果文档
3. 保存为 `final_output.md`

**主持人总结 Prompt**：
```
你是本次讨论的主持人 {agent_name}。

讨论主题：{user_idea}

以下是完整的讨论记录：
{full_discussion_history}

用户的补充意见和约束：
{all_user_feedback}

请综合所有参会者的观点，输出最终结果文档。要求：

1. **核心结论**：明确的结论或推荐方案
2. **关键论据**：支持结论的主要论据（标注来源 AI）
3. **分歧记录**：未达成共识的点及各方立场
4. **行动建议**：下一步可执行的具体建议
5. **风险提示**：讨论中提到的潜在风险

注意：
- 忠实反映讨论内容，不要添加讨论中没有出现的观点
- 对未达成共识的点，保留各方立场而非强行统一
- 突出可执行性，结论要能指导实际行动
```

## 讨论轮次的收敛机制

为避免讨论陷入无限循环，设计以下收敛机制：

1. **主持人主动收敛**：主持人每轮开场时评估讨论是否充分，如果认为已经可以输出结论，在开场引导中建议结束讨论
2. **用户控制**：每轮结束后用户选择是否继续
3. **硬上限**：最大轮次限制（默认 3 轮），到达后自动进入 Phase 3
4. **内容重复检测**：如果本轮发言与上轮高度相似（由主持人判断），建议结束

主持人开场引导中包含一个结构化信号：

```
讨论状态：[CONTINUE] 或 [SUGGEST_CONCLUDE]
理由：...
```

当主持人输出 `[SUGGEST_CONCLUDE]` 时，系统提示用户：

```
💡 主持人建议结束讨论：各方观点已充分表达，继续讨论预计不会产生新的关键洞见。

  [d] 接受建议，生成结果文档
  [c] 我还想继续讨论
```

## 与现有设计的兼容

新的 `discuss` 命令与现有的 `new` / `continue` / `interactive` 命令**并存**：

| 命令 | 模式 | 主持人 | 适用场景 |
|------|------|--------|----------|
| `discuss` | 用户主导 + AI主持 | 用户选定的 AI | 想法讨论、方案设计 |
| `new` + `continue` | 程序编排 | 代码逻辑 | 结构化技术选型 |
| `interactive` | 菜单驱动 | 代码逻辑 | 灵活的多阶段会议 |

用户可以根据场景选择最适合的模式。`discuss` 更适合**开放性讨论**，现有模式更适合**结构化评审**。

## 实现优先级

### P0 — 核心流程

1. `discuss` 命令基本交互框架
2. Phase 1: 独立发言（复用现有并行调用）
3. 主持人选择交互
4. Phase 2: 主持人引导的多轮讨论
5. Phase 3: 主持人生成最终文档

### P1 — 体验优化

6. 观点摘要自动提取（从完整回答中提取关键词和摘要）
7. 主持人收敛建议（`[SUGGEST_CONCLUDE]` 信号）
8. 讨论历史的上下文压缩（复用 `lib/context.py`）

### P2 — 扩展功能

9. `--no-interact` 非交互模式
10. 讨论记录导出（Markdown 报告）
11. 支持中途更换主持人
12. 支持中途增减参会者
